## 1. 项目概述

### 1.1 做了什么？

一个**Web排班管理系统**，用于大学党员工作站的每周值班安排。核心功能：

- 管理员录入成员信息和课表（单周/双周分开）
- 系统根据课表**自动生成**每周值班表
- 支持放假/调休（周末补课）的灵活配置
- 有人请假时自动推荐替补人选
- 统计本学期每人值班时长，导出Excel

### 1.2 为什么做？

**痛点**：党员工作站有50-70人，每周要安排5天×3时段的值班。手工排班需要：
- 逐一确认每个人每个时段有没有课
- 保证公平（不能老排同一个人）
- 遇到调休/放假要手动调整
- 有人请假要找人替

**解决**：自动化这一切，管理员只需要录课表、点一个按钮。

### 1.3 系统规模

| 指标 | 数值 |
|------|------|
| 用户角色 | 1种（管理员，所有人权限相同） |
| 成员规模 | 5主席团 + 12-16部长 + 40-50部员 ≈ 60-70人 |
| 数据库表 | 7张 |
| 前端页面 | 7个（首页/排班/成员/课表/时段/学期/统计） |
| 代码行数 | ~2500行（前端+算法+测试） |
| 测试用例 | 241项，6个测试套件 |

## 2. 需求分析

### 2.1 业务规则（答辩重点）

排班的核心约束条件有6条，按优先级排序：

**规则1：课表驱动（硬约束）**
某人某时段有课 → 绝对不能排。没课 → 可以排。这是排班的物理限制。

**规则2：角色优先级**
部员是主力（占80%时段），部长偶尔参与，主席团仅在人手不足时启用。
- 这个顺序是业务方定的：部员本来就是来值班的，部长有管理职责，主席团是领导。

**规则3：不连续值班**
同一人不会连续两周排班。保证大家都有休息时间。

**规则4：历史公平**
全部历史排班中，次数少的人优先安排。防止有人一直排、有人一直不排。

**规则5：跨周均衡**
课程分单双周，有的人双周课多、单周空闲，有的人相反。算法优先把"另一周课多的人"安排在当前周，把"两周都空的人"留给另一周，实现均衡。

**规则6：每日覆盖 + 自然空缺**
优先保证每天至少1人。实在没人的时段就空着，不强制填充。

### 2.2 单双周概念（答辩高频问题）

大学的课程安排按"单周/双周"区分：
- 单周（第1、3、5、7...周）：上A类课
- 双周（第2、4、6、8...周）：上B类课

所以同一个人的课表在单周和双周是不同的，必须在排班时区分。这就是为什么系统里每个人要录入两份课表。

## 3. 技术选型

### 3.1 技术栈总览

| 层级 | 技术 | 为什么选它 |
|------|------|-----------|
| 前端框架 | React 18 | 组件化开发，状态管理方便，生态成熟 |
| 构建工具 | Vite 5 | 比Webpack快10倍，开发体验好 |
| 路由 | React Router 6 | React生态标准路由方案 |
| 后端 | **没有后端** | 用Supabase BaaS替代 |
| 数据库 | PostgreSQL 15 | 通过Supabase托管 |
| 认证 | Supabase Auth | 内置邮箱+密码认证，开箱即用 |
| 部署 | GitHub Pages | 免费，静态站点托管，自动部署 |

### 3.2 为什么不用传统后端？（答辩必问）

**传统做法**：React前端 → Node.js/Python后端 → PostgreSQL数据库

**我的做法**：React前端 → Supabase JS Client → PostgreSQL数据库

**理由**：
1. **Supabase是BaaS（Backend as a Service）**，它提供了数据库直连的SDK，前端可以直接调用数据库，不需要写后端代码
2. **安全**：Supabase有RLS（Row Level Security），在数据库层面做权限控制，只有登录用户能操作数据
3. **效率**：省去了写API接口、部署服务器的工作量
4. **成本**：Supabase免费额度（500MB数据库+5GB带宽）对这个项目完全够用
5. **适用场景**：这个系统的业务逻辑全在客户端（排班算法在前端跑），没有复杂的服务端计算需求，非常适合BaaS模式

**老师可能的追问**："前端直接操作数据库安全吗？"
**回答**：安全。Supabase的RLS在PostgreSQL层面做行级安全控制。每条SQL执行前PostgreSQL会检查JWT Token，只有authenticated用户能执行CRUD。这不是前端绕过认证，而是认证在数据库层面执行。

### 3.3 为什么选Supabase而不是Firebase？

| 对比 | Supabase | Firebase |
|------|----------|----------|
| 数据库 | PostgreSQL（关系型） | Firestore（NoSQL） |
| 开源 | 是 | 否 |
| SQL支持 | 完整SQL | 有限查询 |
| 价格 | 免费额度更大 | 按量计费 |

这个项目的数据结构（成员、课表、排班）天然是关系型的，需要JOIN查询，PostgreSQL比NoSQL合适得多。

### 4.2 数据流

用户操作 → React组件 → Supabase JS Client → HTTPS请求 → PostgreSQL

以"自动生成排班"为例：

```
用户点"🔄 重新生成"
  → Dashboard.jsx handleGenerate()
    → 从Supabase读取: members, schedules, lastWeek, allAssignments, dayConfig
    → 传入 runSchedulingAlgorithm(params)
    → 算法纯函数计算，返回 { assignments: [...], meta: {...} }
```

关键设计：**排班算法是纯函数**，不依赖数据库、不产生副作用。输入参数 → 计算 → 输出结果。这使得算法可以独立测试（241个用例）。

## 5. 数据库设计

### 5.1 7张表总览

```
members (成员)
  ├── id, name, role, phone, active

course_schedules (课表)
  ├── id, member_id → members.id
  ├── week_type ('单周'/'双周')
  ├── mon_34, mon_67, mon_89  (周一3个时段是否有课)
  ├── tue_34, tue_67, tue_89  (周二)
  ├── ...                      (周三到周日)
  └── UNIQUE(member_id, week_type)  ← 每人每类型最多一条

semester_config (学期配置) — 单例表，只有1行
  ├── id, name, first_week_is_odd, total_weeks, current_week

slot_config (时段人数配置)
  ├── id, day_of_week (1-7), slot ('上午'/'下午1'/'下午2')
  ├── required_count (该时段需要几人)
  └── UNIQUE(day_of_week, slot)

assignments (排班结果) — 核心表
  ├── id, week_number, day_of_week, slot, member_id → members.id
  ├── is_emergency, status ('正常'/'请假')
  ├── leave_next_week (是否下周需要补排)

day_config (工作日配置) — 调休/放假
  ├── id, week_number, day_of_week (1-7)
  ├── is_workday (true=上班, false=放假)
  ├── substitute_for (旧: 补周几的课，向前兼容)
  ├── substitute_for_odd (新: 单周补周几)
  ├── substitute_for_even (新: 双周补周几)
  └── UNIQUE(week_number, day_of_week)

duty_stats (值班统计) — 缓存表
  ├── id, member_id, week_number, total_hours, leave_hours
```

### 5.2 关键设计决策

**为什么course_schedules用列存储而不是行存储？**

行存储方案（不推荐）：
```
member_id | week_type | day | slot | has_class
m1        | 单周      | 周一 | 上午 | true
```
每人的课表需要 7天×3时段×2周型 = 42行，60人需要2520行。

列存储方案（实际使用）：
```
member_id | week_type | mon_34 | mon_67 | mon_89 | tue_34 | ...
m1        | 单周      | true   | false  | true   | false  | ...
```
每人的课表只需要2行（单周1行+双周1行）。

**优势**：
- 行数少60倍
- 一次查询拿到一个人的完整课表（单行），无需JOIN
- 前端网格渲染直接用字段值，无需分组聚合

**为什么day_config有substitute_for, substitute_for_odd, substitute_for_even三个列？**

这是版本迭代的结果：
- `substitute_for`：初版，一个调休日只能设一个补课映射
- `substitute_for_odd` + `substitute_for_even`：改进版，支持单双周独立映射

保留旧列是为了向后兼容——已经存了的数据不会丢，代码用 `||` 做fallback：
```js
const sub = weekType === '单周'
  ? (v.substituteForOdd || v.substituteFor)   // 先取新的，没有再取旧的
  : (v.substituteForEven || v.substituteFor)
```

### 5.3 RLS安全策略

每一张表都启用了Row Level Security，并创建了4条策略（SELECT/INSERT/UPDATE/DELETE）：

```sql
ALTER TABLE members ENABLE ROW LEVEL SECURITY;
CREATE POLICY "允许认证用户读取成员" ON members
  FOR SELECT TO authenticated USING (true);
```

`TO authenticated` 意味着只有登录用户（持有JWT Token）能操作，未登录用户直接被PostgreSQL拒绝。这不是应用层的检查，而是数据库引擎层面的检查。

## 6. 核心算法：排班算法逐行讲解

### 6.1 算法概述

文件：[frontend/src/lib/scheduling-algorithm.js](frontend/src/lib/scheduling-algorithm.js)

**输入**：11个参数
**输出**：`{ assignments: [...] , meta: {...} }`
**性质**：纯函数，不产生副作用

### 6.2 算法参数表

| 参数 | 类型 | 说明 |
|------|------|------|
| members | Array | 所有活跃成员 |
| schedules | Array | 当前周类型的课表（单周或双周） |
| slotConfig | Object | `{"1_上午": 1, "1_下午1": 1, ...}` |
| weekNumber | number | 第几周 |
| lastWeek | Array | 上周排班（防止连续排同一人） |
| allAssignments | Array | 全部历史排班（计算公平性） |
| makeUpMembers | Array | 需要补排的人（之前请假了） |
| otherWeekSchedules | Array | 另一周类型的课表（跨周均衡用） |
| dayConfig | Object | 工作日配置（哪几天上班、调休映射） |
| weekType | string | `'单周'` 或 `'双周'`（决定用哪个调休映射） |

### 6.3 算法流程（两阶段）

#### 阶段0：初始化

```
1. 从dayConfig推导workdays列表（本周哪些天要排班）
2. 为每个调休日建立映射表 dayKeyMap
   - 普通日: 用当天的课表 key（周一→mon, 周二→tue...）
   - 调休日: 用被补日的课表 key（周六补周一→mon）
3. 统计每人历史排班次数 (historyCount)
4. 计算角色配额:
   - 部员≥5人时, 部长配额 = max(1, min(3, ceil(部长人数×0.2)))
   - 部员<5人时, 主席团启用
5. maxPerWeek = max(5, floor(总人数/2))
   - 最少5条，最多不超过一半的人（保证有人休息）
```

#### 角色配额的计算逻辑（答辩重点）

```js
let buZhangQuota = 0
if (buZhangTotal > 0) {
  if (buYuanTotal >= 5) {
    // 部员够用 → 部长只是辅助，每周最多3人，通常为部长人数的20%
    buZhangQuota = Math.max(1, Math.min(3, Math.ceil(buZhangTotal * 0.2)))
  } else {
    // 部员不够 → 部长当主力，至少5人
    buZhangQuota = Math.max(5, Math.ceil(buZhangTotal / 2))
  }
}

// 主席团启用条件：部员+部长总数 < 5
const needZhuXi = (buYuanTotal + buZhangTotal < 5)
const zhuXiQuota = needZhuXi ? Math.max(1, Math.ceil(zhuXiTotal / 2)) : 0
```

这是一个**阶梯式角色策略**：
- 第一层：部员够 → 部员主力
- 第二层：部员不够 → 部长顶上
- 第三层：都不够 → 主席团顶上

#### 阶段1：每日覆盖（保证每天至少1人）

```
for 最多5轮:
  找出还没安排人的工作日
  按"已安排人数"升序排列（缺人的优先）
  对每个缺人的天:
    从候选人池中找该天任一时段有空的人
    安排到第一个匹配的时段
```

**为什么分两个阶段？**
阶段1保证覆盖（每天有人），阶段2保证填充（每个时段有人）。如果不分阶段，可能某天排了3个时段但另一天一个人都没有。

#### 阶段2：轮询补充（填满各时段）

```
while 还有空位 AND 还没达到maxPerWeek上限:
  找出最缺人的工作日和时段
  筛选候选人: 该时段没课的人
  安排第一个人
```

#### 候选人排序算法（getCandidates函数）

这是算法的核心——如何从候选池中选"最合适"的人：

```
排序规则（按优先级从高到低）：
1. 补排人员优先：上周请假的人必须优先安排
2. 配额优先（阶段2）：如果部长还没达到周配额，优先选部长
3. 跨周均衡：另一周空闲越少的人越优先
   - 含义：一个人双周课很多，就应该在单周值班；双周四处都空的人留给双周
4. 历史公平：历史排班次数少的优先
5. 角色权重：部员(0) > 部长(1) > 主席团(2)
```

#### 连续值班约束

```js
if (lastWeekIds.has(m.id)) return false  // 上周排过的，本周不排
```

这个约束在一阶段严格执行。但如果无人可选，算法自然接受空位——这就是"自然空缺"原则。

#### maxPerWeek的含义

```js
const maxPerWeek = Math.max(5, Math.floor(totalMembers / 2))
```

| 总人数 | maxPerWeek | 含义 |
|:--:|:--:|------|
| 10 | 5 | 最多5人值班（5天各1人） |
| 20 | 10 | 最多10人次 |
| 60 | 30 | 最多30人次 |

这个公式保证：至少安排5人（覆盖5天），但不超过一半的人（让另一半休息）。

### 6.4 resolveScheduleKey函数

```js
export function resolveScheduleKey(dayOfWeek, dayConfig, weekType) {
  if (!dayConfig) return ALL_DAY_KEYS[dayOfWeek - 1]
  const v = dayConfig[dayOfWeek]
  if (typeof v === 'object' && v !== null) {
    const sub = weekType === '单周'
      ? (v.substituteForOdd || v.substituteFor)   // 单周先取新列，fallback旧列
      : (v.substituteForEven || v.substituteFor)   // 双周同理
    if (sub && sub >= 1 && sub <= 5) return ALL_DAY_KEYS[sub - 1]
  }
  return ALL_DAY_KEYS[dayOfWeek - 1]
}
```

**功能**：把"今天星期几"映射为"用哪天的课表"

| 输入 | 输出 | 说明 |
|------|------|------|
| day=1, 无dayConfig | `'mon'` | 周一用周一的课表 |
| day=6, substituteForOdd=1 | `'mon'` | 周六补周一，用周一课表 |
| day=6, substituteForOdd=1, substituteForEven=3 | 单周→`'mon'`, 双周→`'wed'` | 单双周不同课 |

## 7. 调休系统（最复杂的业务逻辑）

### 7.1 业务场景

大学的调休场景：
- **放假**：清明节周三放假 → 当天不值班
- **补课**：周六补周一的课 → 周六值班，但检查课表冲突时要用"周一"的课表

### 7.2 为什么讲这是最复杂的部分？

因为调休涉及4个维度的联动：

```
调休配置 (DaySelector UI)
  → 数据库存储 (day_config表, 3个substitute列)
    → 算法读取 (resolveScheduleKey, weekType参数)
      → 排班生成 (用映射后的课表key检查冲突)
        → 请假替补 (也要用正确的课表key)
```

任何一个环节出错都会导致排班错误。

### 7.3 DaySelector组件设计

### 8.4 防重复生成机制

**问题**：React 18开发模式下StrictMode会导致useEffect执行两次，可能触发两次排班生成，导致重复数据。

**解决方案（三层防护）**：
```js
const generatingRef = useRef(false)  // 互斥锁

async function handleGenerate() {
  if (generatingRef.current) return   // 1. 检查锁，已锁则跳过
  generatingRef.current = true         // 2. 加锁
  setGenerating(true)                  // 3. UI显示加载中（按钮disabled）
  try {
    // ... 执行排班 ...
  } finally {
    setGenerating(false)
    generatingRef.current = false     // 4. 无论成功失败都释放锁
  }
}
```

三层防护：
1. `generatingRef` 互斥锁（内存级，防止同一组件实例重复调用）
2. `generating` 状态控制按钮disabled（UI级，防止用户连点）
3. 移除了 `<StrictMode>`（框架级，防止开发环境双重挂载）

## 9. 关键技术难点与解决方案

### 9.1 难点一：调休日课表映射

**问题**：周六补周一的课，但课表是按单双周分的，单周周一和双周周一可能不同。

**解决**：引入 `weekType` 参数贯穿全链路 + 双列存储 `substituteForOdd/substituteForEven`。

**影响范围**：DaySelector UI → 数据库 → 算法 resolveScheduleKey → 排班生成 → 请假替补，共5个环节6个文件。

### 9.2 难点二：跨周均衡的量化

**问题**：怎么判断一个人应该安排在单周还是双周？

**解决**：计算每个人在"另一周类型"的空闲时段数（otherWeekFreeCount），空闲越少越优先安排在当前周。

```js
// 双周只有1个时段空闲的人 → 必须安排在单周
// 双周21个时段全空的人 → 留给双周，单周不优先安排
```

### 9.3 难点三：并发重复排班

**问题**：切换到新周后每个时段出现两个相同的人。

**根因**：`<StrictMode>` + `useEffect` 导致两次并发 `autoGenerate`，第一次 delete 完成、第一次 insert 开始时，第二次 delete 也开始了，时序打乱。

**解决**：移除 StrictMode + useRef互斥锁 + UI disabled三层防护。

### 9.4 难点四：调休toggle与下拉框的竞态

**问题**：用户开启周末调休后立即修改下拉框，toggle的异步保存可能还没完成，changeSubstitution查不到记录就静默跳过。

**解决**：changeSubstitution增加 else 分支——如果查不到记录就插入新记录（Upsert语义）。

### 10.2 等价类测试的设计思路（答辩重点）

等价类划分是一种黑盒测试方法，将输入数据划分为若干等价类，每个类中的值对程序行为的影响是等价的。

我为系统划分了8个维度：

| 维度 | 等价类数 | 每个类代表什么 |
|------|:--:|------|
| resolveScheduleKey | 8 | null/boolean/旧substituteFor/仅odd/仅even/混合/边界值/工作日行为 |
| DayConfig→workdays | 5 | 无配置/放假/调休/全开/全关fallback |
| 角色配额 | 4 | 仅部员(≥5)/部员+部长/不足5人(主席团)/仅部长 |
| 课表冲突 | 3 | 全空/全满/混合(半有课半无课) |
| SlotConfig | 4 | 默认/周末0/双人时段/全0工作日保护 |
| 调休单双周独立 | 4 | 仅odd/仅even/同值/异值 |
| 边界条件 | 6 | 空成员/全排过/maxPerWeek/补排/历史公平/跨周均衡 |
| 逻辑一致性 | 6 | 无重复/每人1次/冲突排除/调休排除/status/weekType传递 |

**核心思想**：不需要对每个具体值都测试，只需要测试每个等价类中的一个代表值。比如"课表映射"这个维度，测试了 1→mon, 3→wed, 5→fri 三种就够了，不需要测 2→tue 和 4→thu。

## 12. 答辩常见问题模拟

### Q1: "你为什么要做这个系统？"

**回答思路**：痛点驱动。党员工作站50-70人手工排班效率低、易出错、不公平。我观察到这个需求，开发了一套自动化排班系统，让管理员只需要录课表点按钮即可。

### Q2: "系统架构为什么选择BaaS模式？"

**回答**：这个系统的特点是业务逻辑集中在客户端（排班算法在浏览器里跑），没有复杂的服务端计算需求。BaaS模式省去了编写后端API和部署服务器的工作，Supabase提供的PostgreSQL+Auth+RLS正好满足需求。而且免费额度足够，项目可以永久运行。

### Q3: "排班算法的核心思路是什么？"

**回答**：两阶段贪心算法。阶段1保证每天至少1人覆盖，阶段2轮询填充剩余空位。候选人排序考虑5个维度：补排优先→配额控制→跨周均衡→历史公平→角色权重。算法是纯函数，241个测试用例保证正确性。

### Q4: "调休功能怎么实现的？"

**回答**：核心是 `resolveScheduleKey` 函数，它把"今天星期几"映射为"用哪天的课表检查冲突"。数据库用单双周两个独立字段存储映射关系，前端用双下拉框让管理员分别设置。整个链路涉及UI组件→数据库→算法→排班生成→请假替补5个环节。

### Q5: "你怎么保证系统安全？"

**回答**：三层安全机制：① Supabase Auth管理用户登录和JWT Token签发；② PostgreSQL RLS在数据库引擎层面控制权限，只有持有效JWT的authenticated用户能读写数据；③ 环境变量存储Supabase URL和API Key，不暴露在代码中。

### Q6: "系统能支撑多少人？"

**回答**：数据库设计支持60-70人。排班算法每多10人增加约50ms计算时间（纯内存计算），完全可以接受。Supabase免费额度对这个规模绰绰有余。

### Q7: "如果有人在两个浏览器同时点生成排班会怎样？"

**回答**：后点的那个会覆盖前一个的结果。因为系统对本周排班是先delete再insert，最后一次操作的结果是最终结果。这是一个可接受的设计——同一时刻只有一个管理员在操作排班是合理的假设。代码用互斥锁防止了同一浏览器内的并发问题。

### Q8: "为什么不支持手机端？"

**回答**：功能上是支持的（响应式布局），但课表录入的网格交互在手机上体验不好。这个系统的核心操作场景是管理员在电脑前处理排班，手机端不做专门优化是合理的取舍。

### Q9: "241个测试用例怎么设计的？"

**回答**：分层设计。基础算法27项；调休场景32项；大规模综合49项；全学期模拟14项；功能全覆盖59项；等价类测试60项覆盖8个维度。等价类方法可以在有限用例内达到最高覆盖率——每个等价类只需要一个代表值。

### Q10: "系统有什么可以改进的地方？"

**回答**：
1. 目前没有操作日志，管理员误操作后无法追溯
2. 课表录入需要逐个成员操作，60人效率较低（可以加批量模板）
3. 没有邮件/微信通知功能（值班提醒）
4. 排班算法可以考虑更多的优化目标（如相邻周同一时段尽量安排不同的人）
这些在后续版本中可以逐步完善。
