### 第2步：数据库设计与创建

#### 2.1 设计思路（面试重点）

**第一步：画ER图（用脑子画就行）**

```
members (成员)
  │
  ├──< course_schedules (课表) — 每人2行（单周+双周）
  │
  ├──< assignments (排班结果) — 核心表，记录谁在哪天哪个时段值班
  │
semester_config (学期配置) — 单例，只有1行
slot_config (时段人数配置) — 每天每时段需要几人
day_config (工作日配置) — 调休/放假标记
duty_stats (值班统计) — 缓存表（可选）
```

**设计原则**：
1. 每张表只做一件事（单一职责）
2. 用外键（REFERENCES）维护数据一致性
3. 用 UNIQUE 约束防止重复数据
4. 用 CHECK 约束限制字段值范围

#### 2.2 核心设计决策：列存储 vs 行存储

**这是面试必问题**，你需要能讲清楚：

**方案A（行存储 — 不推荐）**：
```sql
-- 每人的课表需要 7天×3时段×2周型 = 42行！
member_id | week_type | day | slot   | has_class
m1        | 单周      | 周一 | 上午   | true
m1        | 单周      | 周一 | 下午1  | false
-- ... 60人×42行 = 2520行
```

**方案B（列存储 — 实际采用）**：
```sql
-- 每人只需要2行（单周1行+双周1行）
member_id | week_type | mon_34 | mon_67 | mon_89 | tue_34 | ...
m1        | 单周      | true   | false  | true   | false  | ...
```

**选择方案B的理由**：
1. **行数少60倍**：60人只需要120行，不是2520行
2. **读取效率**：`SELECT * WHERE member_id = ? AND week_type = ?` 一次拿到完整课表
3. **前端渲染方便**：直接取字段值 `schedule.mon_34`，不需要分组聚合
4. **唯一约束简单**：`UNIQUE(member_id, week_type)` 即可

**什么时候用行存储？**
- 如果时段数量会动态变化（这个项目不会，固定21个时段）
- 如果需要按课表字段做复杂查询（这个项目不需要，每次只是"查某人的课表"）

#### 2.3 完整建表SQL（逐句讲解）

```sql
-- ===== 1. 成员表 =====
CREATE TABLE IF NOT EXISTS members (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  -- ↑ UUID做主键的优势：分布式生成不冲突，比自增ID安全
  name        TEXT NOT NULL,
  role        TEXT NOT NULL CHECK (role IN ('部员', '部长', '主席团')),
  -- ↑ CHECK约束：数据库层面防止输错角色名
  phone       TEXT DEFAULT '',
  active      BOOLEAN DEFAULT true,
  -- ↑ 软删除：标记为false而非真删除，保留历史数据
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- ===== 2. 课表表 =====
CREATE TABLE IF NOT EXISTS course_schedules (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id   UUID REFERENCES members(id) ON DELETE CASCADE,
  -- ↑ ON DELETE CASCADE：删除成员时自动删除其课表，防止孤儿数据
  week_type   TEXT NOT NULL CHECK (week_type IN ('单周', '双周')),
  -- 周一3个时段（34=第3-4节，67=第6-7节，89=第8-9节）
  mon_34      BOOLEAN DEFAULT false,  -- true=有课不能值班
  mon_67      BOOLEAN DEFAULT false,
  mon_89      BOOLEAN DEFAULT false,
  -- 周二到周日同理（共21个布尔字段）
  -- ...
  UNIQUE(member_id, week_type)
  -- ↑ 每人每类型只能有一条课表记录
);

-- ===== 3. 学期配置表（单例表，只有1行） =====
CREATE TABLE IF NOT EXISTS semester_config (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name              TEXT NOT NULL DEFAULT '新学期',
  first_week_is_odd BOOLEAN DEFAULT true,
  -- ↑ 关键字段：决定如何判断单双周
  total_weeks       INTEGER DEFAULT 20,
  current_week      INTEGER DEFAULT 1,
  current_mode      TEXT DEFAULT '一般' CHECK (current_mode IN ('一般', '紧急')),
  created_at        TIMESTAMPTZ DEFAULT now()
);

-- ===== 4. 时段人数配置 =====
CREATE TABLE IF NOT EXISTS slot_config (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  day_of_week     INTEGER NOT NULL CHECK (day_of_week BETWEEN 1 AND 7),
  slot            TEXT NOT NULL CHECK (slot IN ('上午', '下午1', '下午2')),
  required_count  INTEGER DEFAULT 1,
  UNIQUE(day_of_week, slot)
);

-- ===== 5. 排班结果（核心表） =====
CREATE TABLE IF NOT EXISTS assignments (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  week_number       INTEGER NOT NULL,
  day_of_week       INTEGER NOT NULL CHECK (day_of_week BETWEEN 1 AND 7),
  slot              TEXT NOT NULL CHECK (slot IN ('上午', '下午1', '下午2')),
  member_id         UUID REFERENCES members(id) ON DELETE CASCADE,
  is_emergency      BOOLEAN DEFAULT false,
  status            TEXT DEFAULT '正常' CHECK (status IN ('正常', '请假')),
  leave_next_week   BOOLEAN DEFAULT false,
  -- ↑ 请假标记：下周需要补排此人
  created_at        TIMESTAMPTZ DEFAULT now()
);

-- ===== 6. 值班统计（缓存表，也可实时计算） =====
CREATE TABLE IF NOT EXISTS duty_stats (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id     UUID REFERENCES members(id) ON DELETE CASCADE,
  week_number   INTEGER NOT NULL,
  total_hours   REAL DEFAULT 0,
  leave_hours   REAL DEFAULT 0,
  UNIQUE(member_id, week_number)
);

-- ===== 7. 工作日配置（调休/放假） =====
CREATE TABLE IF NOT EXISTS day_config (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  week_number       INTEGER NOT NULL,
  day_of_week       INTEGER NOT NULL CHECK (day_of_week BETWEEN 1 AND 7),
  is_workday        BOOLEAN DEFAULT true,
  substitute_for    INTEGER DEFAULT NULL,    -- 补周几的课（旧版）
  substitute_for_odd  INTEGER DEFAULT NULL,  -- 单周补周几（新版）
  substitute_for_even INTEGER DEFAULT NULL,  -- 双周补周几（新版）
  UNIQUE(week_number, day_of_week)
);
```

#### 2.4 RLS安全策略

```sql
-- 对每张表启用行级安全
ALTER TABLE members ENABLE ROW LEVEL SECURITY;

-- 创建策略：只有登录用户（authenticated）能操作
CREATE POLICY "允许认证用户读取成员" ON members
  FOR SELECT TO authenticated USING (true);
CREATE POLICY "允许认证用户插入成员" ON members
  FOR INSERT TO authenticated WITH CHECK (true);
CREATE POLICY "允许认证用户更新成员" ON members
  FOR UPDATE TO authenticated USING (true);
CREATE POLICY "允许认证用户删除成员" ON members
  FOR DELETE TO authenticated USING (true);

-- 其他6张表同样创建4条策略...
```

**RLS工作原理**：
1. 用户登录 → Supabase Auth 签发 JWT Token
2. 前端调用 `supabase.from('members').select('*')` → JS Client 自动在请求头携带 JWT
3. PostgreSQL 在执行 SQL 前检查 JWT → 验证是否 `authenticated`
4. 未登录用户的所有操作直接被数据库拒绝（不是前端拦截的，是数据库层的安全检查）

**面试时这样说**："RLS 把安全检查下沉到数据库引擎层面，即使有人绕过前端直接发HTTP请求，PostgreSQL也会拒绝未认证的请求。这比在后端代码里写 `if (!user) return 403` 更安全，因为无法绕过。"

### 第3步：前端骨架搭建

#### 3.1 Supabase 客户端配置

```js
// src/lib/supabase.js
import { createClient } from '@supabase/supabase-js'

const supabaseUrl = import.meta.env.VITE_SUPABASE_URL
const supabaseAnonKey = import.meta.env.VITE_SUPABASE_ANON_KEY
```

#### 7.4 核心函数1：getCandidates — 候选人排序

这是算法的灵魂：

```js
function getCandidates(extraFilter, phase = 1) {
  // 第一步：筛选
  let pool = members.filter(m => {
    if (thisWeekAssigned.has(m.id)) return false     // 本周已排过
    if (lastWeekIds.has(m.id)) return false           // 上周排过
    if (m.role === '主席团' && !needZhuXi) return false // 主席团禁用中
    if (超过该角色本周配额) return false                // 配额已满
    if (extraFilter && !extraFilter(m)) return false   // 额外过滤
    return true
  })

  // 第二步：排序（5个维度，优先级从高到低）
  pool.sort((a, b) => {
    // 维度1: 补排人员优先（上周请假的人）
    //   补排 → 0, 正常 → 1  → 补排的排前面
    if (aIsMakeUp !== bIsMakeUp) return aIsMakeUp - bIsMakeUp

    // 维度2: 配额优先（仅阶段2）
    //   部长还没达到周配额 → 优先于部员
    if (phase === 2 && buZhangGap > 0) {
      if (aIsZhang !== bIsZhang) return aIsZhang - bIsZhang
    }

    // 维度3: 跨周均衡
    //   另一周空闲越少的人越优先
    //   含义: 双周课多的人→安排在单周，双周空闲留给双周用
    if (aOtherFree !== bOtherFree) return aOtherFree - bOtherFree

    // 维度4: 历史公平
    //   历史排班次数少的优先
    if (aHistory !== bHistory) return aHistory - bHistory

    // 维度5: 角色权重
    //   部员(0) > 部长(1) > 主席团(2)
    return ROLE_WEIGHT[a.role] - ROLE_WEIGHT[b.role]
  })

  return pool
}
```

**面试时怎么讲这个排序**：

"候选人排序有5个维度，按优先级依次是：
1. **补排优先**：上周请假的人本周优先安排，保证公平
2. **配额控制**：部长每周有配额上限，配额内优先选部长
3. **跨周均衡**：计算每个人在另一周的课表空闲程度，双周课多的人优先安排在单周
4. **历史公平**：本学期值班次数少的优先
5. **角色权重**：部员 > 部长 > 主席团

每个维度只在上一维度相同时才起作用，这是一个优先级递减的多级排序。"

#### 7.5 核心函数2：resolveScheduleKey — 调休映射

```js
export function resolveScheduleKey(dayOfWeek, dayConfig, weekType) {
  // 没有 dayConfig → 返回当天的 key
  if (!dayConfig) return ALL_DAY_KEYS[dayOfWeek - 1]

  const v = dayConfig[dayOfWeek]

  // 新格式（rich object）
  if (typeof v === 'object' && v !== null) {
    // 按周类型选择: 单周用 substituteForOdd, 双周用 substituteForEven
    const sub = weekType === '单周'
      ? (v.substituteForOdd || v.substituteFor)    // || 是fallback
      : (v.substituteForEven || v.substituteFor)
    if (sub && sub >= 1 && sub <= 5) return ALL_DAY_KEYS[sub - 1]
  }

  // 默认：当天用当天的课表
  return ALL_DAY_KEYS[dayOfWeek - 1]
}
```

| 输入 | weekType | 返回 | 说明 |
|------|------|------|------|
| day=1, 周一 | 任意 | `'mon'` | 周一用周一课表 |
| day=6, substituteForOdd=1 | 单周 | `'mon'` | 周六单周补周一 |
| day=6, substituteForOdd=1, substituteForEven=3 | 双周 | `'wed'` | 周六双周补周三 |
| day=7, 无dayConfig | - | `'sun'` | 周日用周日课表 |

#### 7.7 算法关键数值设计

```
角色配额公式:

情况A: 部员 ≥ 5 人（正常情况，最常见）
  部长配额 = max(1, min(3, ceil(部长人数 × 0.2)))
  含义: 部长只是辅助，每周1-3人参与

  例: 16个部长 → ceil(16×0.2) = ceil(3.2) = 4 → min(3,4) = 3 → max(1,3) = 3
      每周最多3个部长值班

情况B: 部员 < 5 人（人员不足）
  部长配额 = max(5, ceil(部长人数/2))
  主席团配额 = max(1, ceil(主席团人数/2))
  含义: 部长当主力，主席团也启用


每周总上限:

  maxPerWeek = max(5, floor(总人数/2))

  例: 60人 → floor(30) = 30 → max(5, 30) = 30
      每人每周最多排1次，30个人值这周的班，另外30人休息
      下周优先排另外30人 (跨周均衡)

  例: 5人 → floor(2.5) = 2 → max(5, 2) = 5
      下限是5，保证工作日每天至少有1人的可能性
```

#### 8.2 自动生成排班的完整流程

```js
async function autoGenerate(sem, scMap) {
  // 0. 防重复生成锁
  if (generatingRef.current) return
  generatingRef.current = true
  setGenerating(true)

  try {
    const cw = sem.current_week || 1

    // 1. 计算本周是单周还是双周
    const weekType = (sem.first_week_is_odd ? cw % 2 === 1 : cw % 2 === 0)
      ? '单周' : '双周'
    const otherWeekType = weekType === '单周' ? '双周' : '单周'

    // 2. 从数据库读取所有需要的参数
    const { data: members } = await supabase.from('members').select('*').eq('active', true)
    const { data: schedules } = await supabase.from('course_schedules').select('*').eq('week_type', weekType)
    const { data: otherWeekSchedules } = await supabase.from('course_schedules').select('*').eq('week_type', otherWeekType)
    const { data: lastWeek } = await supabase.from('assignments').select('member_id').eq('week_number', cw - 1)
    const { data: allAssignments } = await supabase.from('assignments').select('member_id').eq('status', '正常')
    const { data: makeUpMembers } = await supabase.from('assignments').select('member_id').eq('leave_next_week', true)

    // 3. 调用纯函数算法
    const result = runSchedulingAlgorithm({
      members, schedules, slotConfig: scMap, weekNumber: cw,
      lastWeek, allAssignments, makeUpMembers,
      otherWeekSchedules, dayConfig, weekType
    })

    // 4. 写入数据库（先删后插）
    if (result.assignments.length > 0) {
      await supabase.from('assignments').delete().eq('week_number', cw)  // 删除本周旧排班
      await supabase.from('assignments').insert(result.assignments)       // 插入新排班

      // 5. 清除补排标记
      if (makeUpMembers && makeUpMembers.length > 0) {
        await supabase.from('assignments').update({ leave_next_week: false })
          .eq('leave_next_week', true)
      }
    }

    // 6. 刷新显示
    loadCurrentWeek()
    showToast(`本周排班已生成：${result.assignments.length} 条`, 'success')

  } finally {
    // 无论如何都要释放锁
    setGenerating(false)
    generatingRef.current = false
  }
}
```

#### 8.3 请假处理流程

```
管理员点"请假"
  → 确认弹窗
    → update: status='请假', leave_next_week=true
      → 查询该时段所有没课的人
        → 排序: 空课表优先 → 部员优先
          → 弹窗显示候选人列表
            → 管理员选一个人
              → insert: 新排班记录（替补）
                → 下周排班时，被请假者自动补排
```

```js
async function handleLeave(assignment) {
  // 1. 确认
  if (!confirm(`确定将 ${assignment.members?.name} 标记为请假吗？`)) return

  // 2. 标记请假
  await supabase.from('assignments').update({
    status: '请假',
    leave_next_week: true  // 关键：下周补排标记
  }).eq('id', assignment.id)

  // 3. 找替补候选人
  //    resolveScheduleKey: 用正确的课表映射（支持调休）
  const dKey = resolveScheduleKey(assignment.day_of_week, dayConfig, weekType)
  const sKey = SLOT_KEYS[SLOTS.indexOf(assignment.slot)]
  const colKey = `${dKey}_${sKey}`  // 如: "mon_34"

  // 4. 筛选该时段没课的人
  const available = members.filter(m => {
    if (m.id === assignment.member_id) return false  // 排除本人
    const s = scheduleMap[m.id]
    if (!s) return true           // 没有课表 → 完全空闲 → 优先
    return !s[colKey]             // 该时段没课
  })

  // 5. 排序：完全没课表的排最前面（🟢空课表标记）
  available.sort((a, b) => {
    const aNoSchedule = scheduleMap[a.id] ? 1 : 0  // 没课表 → 0（排前面）
    const bNoSchedule = scheduleMap[b.id] ? 1 : 0
    if (aNoSchedule !== bNoSchedule) return aNoSchedule - bNoSchedule
    return ROLE_WEIGHT[a.role] - ROLE_WEIGHT[b.role]
  })

  // 6. 弹窗让管理员选择替补
  setCandidates(available)
  setShowReplace(assignment)
}
```

### 第9步：调休系统

#### 9.1 业务背景

**Toggle 逻辑**：
```js
async function toggle(day) {
  const current = dayConfig[day]
  const wasWorkday = typeof current === 'object' ? current.isWorkday : current

  const newValue = {
    isWorkday: !wasWorkday,
    substituteForOdd: null,
    substituteForEven: null
  }

  // 如果开启且是周末(6或7)，自动设置默认补课映射
  if (!wasWorkday && (day === 6 || day === 7)) {
    // 找第一个工作日作为默认值
    const firstWorkday = [1,2,3,4,5].find(d => {
      const v = dayConfig[d]
      return typeof v === 'object' ? v.isWorkday : true
    })
    if (firstWorkday) {
      newValue.substituteForOdd = firstWorkday
      newValue.substituteForEven = firstWorkday
    }
  }

  // Upsert 到数据库
  const { data: existing } = await supabase.from('day_config')
    .select('id').eq('week_number', weekNumber).eq('day_of_week', day)
    .maybeSingle()

  if (existing) {
    await supabase.from('day_config').update(newValue).eq('id', existing.id)
  } else {
    await supabase.from('day_config').insert({
      week_number: weekNumber, day_of_week: day, ...newValue
    })
  }
}
```

**ChangeSubstitution 逻辑（含竞态修复）**：
```js
async function changeSubstitution(day, type, value) {
  // type 是 'odd' 或 'even'
  const field = type === 'odd' ? 'substituteForOdd' : 'substituteForEven'
  const update = { [field]: value }

  const { data: existing } = await supabase.from('day_config')
    .select('id').eq('week_number', weekNumber).eq('day_of_week', day)
    .maybeSingle()

  if (existing) {
    await supabase.from('day_config').update(update).eq('id', existing.id)
  } else {
    // 关键修复：如果 toggle 还没保存完，直接插入完整记录
    await supabase.from('day_config').insert({
      week_number: weekNumber, day_of_week: day,
      isWorkday: true,
      substituteForOdd: type === 'odd' ? value : null,
      substituteForEven: type === 'even' ? value : null,
    })
  }
}
```

#### 12.5 等价类测试的核心思想

等价类划分 = 把无穷多种可能的输入分成有限的几类，每类里选一个代表测试。

```
不是测试"补周一、补周二、补周三、补周四、补周五"5种映射，
而是把它们看作一个等价类，只测试一种就够了。

但"有映射 vs 无映射"是不同等价类，必须分别测试。
```

**8个维度**：
1. resolveScheduleKey（8类）：null / boolean / 旧substituteFor / 仅odd / 仅even / 混合 / 边界值 / 工作日行为
2. DayConfig→workdays（5类）：无配置 / 放假 / 调休 / 全开 / 全关fallback
3. 角色配额（4类）：仅部员 / 部员+部长 / 不足5人 / 仅部长
4. 课表冲突（3类）：全空闲 / 全满 / 混合
5. SlotConfig（4类）：默认 / 周末0 / 双人时段 / 全0保护
6. 调休单双周独立（4类）：仅odd / 仅even / 同值 / 异值
7. 边界条件（6类）：空成员 / 全排过 / maxPerWeek / 补排 / 历史公平 / 跨周均衡

## Part C：实际踩坑记录

> 这些是你面试时最亮眼的素材——"我遇到过什么问题，怎么解决的"。

### C.1 并发重复排班

**现象**：切换到新周后，每个时段出现两个相同的人。

**排查过程**：
1. 检查数据库 → 同一时段确实有两条记录
2. 检查插入代码 → 逻辑是 delete 全部 + insert 新数据，看起来没问题
3. 加 console.log → 发现 delete 被调用了两次
4. 追溯到 `useEffect` → 被触发了两次
5. 根因：`<StrictMode>` 在 React 18 开发模式下故意双重执行 useEffect

**时序分析**：
```
时间线  →
useEffect #1: delete → insert(30条) —— 正常
useEffect #2: delete → insert(30条) —— 因为 StrictMode 又执行了一次
实际: delete→insert→delete→insert → 第二次 insert 前第一次的数据还没删完，结果翻倍
```

**解决方案（三层防护）**：
```js
// 1. 移除 StrictMode
// 2. 添加互斥锁
const generatingRef = useRef(false)

async function autoGenerate() {
  if (generatingRef.current) return   // 已锁，跳过
  generatingRef.current = true        // 加锁
  try {
    // ... 排班逻辑 ...
  } finally {
    generatingRef.current = false     // 无论成功失败都解锁
  }
}
// 3. UI 层 disabled 按钮
```

### C.2 DaySelector 竞态条件

**现象**：用户开启周末调休后立即修改下拉框，数据丢失。

**根因**：JavaScript 异步操作的时间顺序：
```
操作1: click toggle(周六) → 触发 async toggle() → 等待 Supabase 响应...
操作2: change dropdown(周六, odd, 1) → 触发 async changeSubstitution()

但操作2可能比操作1先完成！
changeSubstitution: SELECT day_config WHERE week=X AND day=6 → 找不到记录 → 跳过
toggle: INSERT day_config (is_workday=true) → 保存成功
但是 changeSubstitution 已经跳过了 → substituteForOdd 丢失！
```

**修复**：`changeSubstitution` 增加 upsert 语义：
```js
if (existing) {
  await supabase.from('day_config').update(update).eq('id', existing.id)
} else {
  // 关键：如果记录还不存在，插入完整记录
  await supabase.from('day_config').insert({
    week_number: weekNumber, day_of_week: day,
    isWorkday: true,
    substituteForOdd: type === 'odd' ? value : null,
    substituteForEven: type === 'even' ? value : null,
  })
}
```

### C.3 历史值班表只显示周一至周五

**现象**：调休周周六有排班，但历史值班表里看不到。

**根因**：渲染时用 `[1,2,3,4,5]` 固定5天循环，调休周周六(6)的数据被遗漏。

**修复**：循环改为 `[1,2,3,4,5,6,7]` 全部7天。

### C.4 开发环境 localhost 访问失败

**现象**：`npm run dev` 后打开 `localhost:5173` 显示404或无法连接。

**根因**：`vite.config.js` 的 `base` 写死为 `/party-station-scheduler/`，开发环境下也走了子路径。

**修复**：
```js
base: mode === 'production' ? '/party-station-scheduler/' : '/'
```

### C.5 第二周返回首页白屏

**现象**：当前周设为第2周后，回到首页白屏。

**根因**：Dashboard 组件 `loadAll()` 中引用 `ALL_DAYS_SHORT` 和 `ALL_DAYS` 变量，但这些常量定义在 Dashboard.jsx 文件顶部...等等，实际排查发现这两个变量在特定代码路径下未定义。

**教训**：使用常量前先确认定义位置，跨文件共享的常量应该提取到独立的 `constants.js` 文件。

### C.6 调休周六无人值班

**现象**：周六设为调休日后，排班结果里周六没有任何人。

**根因**：算法的 `slotConfig` 读取时，周六默认 `required_count=0`。`slotConfig` 的默认值是 `{ '6_上午': 0, '6_下午1': 0, ... }`，即使 dayConfig 标记周六为工作日，算法里的 `required` 仍然是 0。

**修复**：两处改动：
1. 算法中 `required = Math.max(required, 1)` 保证工作日最少1人
2. `slotConfig` 默认值中，周末各时段默认为 0，但管理员可以手动调高

## Part D：关键决策与替代方案

> 面试官会问"为什么不选X？"，你需要能讲清楚每项决策的理由。

### D.1 为什么用 BaaS 而不是传统后端？

| 对比维度 | BaaS (Supabase) | 传统后端 (Node.js+Express) |
|------|------|------|
| 开发量 | 零后端代码 | 需要写API接口、部署服务器 |
| 安全性 | RLS在PostgreSQL层 | 需要自己写认证中间件 |
| 成本 | 免费额度内免费 | 需要服务器（至少几十元/月） |
| 维护 | Supabase团队维护 | 自己维护 |
| 适用场景 | 客户端业务逻辑为主 | 复杂服务端计算 |

**选择BaaS的理由**：
1. 排班算法在浏览器里跑（纯函数），不需要服务端计算
2. 数据关系简单（7张表，标准CRUD），不需要复杂的后端逻辑
3. 只有管理员使用（<5人），不需要用户管理、权限分级
4. 目标是永久免费运行

### D.2 为什么课表用列存储？

详见 [第2步 2.2节](#22-核心设计决策列存储-vs-行存储)

### D.3 为什么用单双周双列而不是单列？

**问题**：`substitute_for` 只有一个值，但同一天单周和双周补的课可能不同。

**方案A（单列）**：周六只能设一个补课映射，单双周都走同一个。
→ 如果一个同学单周周一没课但双周周一有课，双周周六值班时会错误地认为他有空。

**方案B（双列）**：`substitute_for_odd` + `substitute_for_even` 分别存储。
→ 算法根据 `weekType` 选择对应列，单双周独立映射，互不干扰。

**向后兼容**：保留了旧列 `substitute_for`，用 `||` 做 fallback：
```js
const sub = weekType === '单周'
  ? (v.substituteForOdd || v.substituteFor)   // 新列优先，旧列兜底
  : (v.substituteForEven || v.substituteFor)
```

### D.4 为什么不支持手机端？

功能上响应式布局支持手机端，但课表录入的 7×3 网格在手机屏幕上体验很差。

这是合理的产品取舍：核心操作场景是管理员在电脑前工作，手机端只是偶尔查看。不专门优化手机端的开发时间可以投入到更有价值的功能上。

### D.5 为什么测试用 Node.js 而不是浏览器？

因为算法是纯函数——输入参数 → 计算 → 输出结果，不依赖 DOM、不依赖浏览器 API。

Node.js 跑测试比浏览器快得多（241个用例几秒跑完），而且可以集成到 CI/CD（GitHub Actions 自动跑测试）。

### E.2 常见追问速答

**Q: 项目最大挑战是什么？**
A: 调休系统的全链路设计。调休涉及 DaySelector UI → 数据库 → resolveScheduleKey → 排班生成 → 请假替补 5个环节的联动，任何一个环节出错都会导致排班错误。特别是单双周独立映射的设计，需要处理新旧格式兼容和竞态条件。

**Q: 你学到了什么？**
A: 三个方面。技术上：React Hooks、PostgreSQL、BaaS架构。工程上：测试驱动设计（纯函数算法可以脱离UI独立测试）。产品上：技术选型要匹配实际场景（高校小团队不需要 Kubernetes，BaaS 足够）。

**Q: 241个测试用例怎么设计的？**
A: 分层设计。基础算法27项验证核心逻辑，调休场景32项覆盖边界情况，大规模综合49项模拟真实环境，全学期14项验证长期运行，全覆盖59项回归验证，等价类60项用科学方法达到最高覆盖率。等价类方法可以在有限用例内覆盖无穷多种输入组合。

**Q: 有什么不足？**
A: 三个可以改进的地方：1)没有操作日志，管理员误操作无法追溯；2)课表录入需要逐个成员操作，可以加批量模板功能；3)排班算法目前是贪心，理论上可以引入回溯找更优解，但当前规模下贪心已经足够好。
